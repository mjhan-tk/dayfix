# dayfix — 서비스 기획 & 정책

> 캘린더 통합 기반의 **사내 회의 일정 조율** 서비스.
> "여러 사람이 모두 괜찮은 시간을 빠르게, 후회 없이 잡는다."

이 문서는 디자인/기획 논의에서 합의한 내용을 정리한 살아있는 문서다. 기획은 자주 바뀐다 — 결정 로그(§7)를 함께 갱신할 것.

---

## 1. 한 줄 정의

같은 회사 동료들이 정해진 기한 안에 1시간짜리 회의를 잡을 때, **서로의 사정을 이해하고 모두에게 괜찮은 시간을 찾아 확정**하도록 돕는 캘린더 통합 서비스.

대표 시나리오: 동료 6명이 다음 주까지 1시간 회의. 누구는 점심 직후를 피하고 싶고, 누구는 특정 요일 외근. 필수 참석자와 선택 참석자가 섞여 있다.

---

## 2. 문제 정의

여러 사람이 함께 일정을 잡는 일은 생각보다 번거롭다. 그 번거로움의 진짜 원인은 "빈 시간 찾기"가 아니다.

### 통찰 ① — 커뮤니케이션 누수 (communication leakage)
도구가 사람들의 *사정/맥락*을 담지 못하면, 그 대화는 결국 **다른 메신저로 새어 나간다.** "나 그날 좀 그런데…"가 카톡으로 빠지는 순간 일정 도구는 무력해진다. 일정 누수의 발원지는 늘 **"누군가의 미응답·미합의"**다.

### 통찰 ② — 후순위의 체면 문제 (the dignity problem)
일정 조율은 logistics가 아니라 **사회적 행위**다. 누군가를 "선택참여"로 빼는 것은 기능이 아니라 *지위 신호*이고, 잘못하면 사람을 다치게 한다. 대부분의 도구가 이 감정·정치 레이어를 무시한다.

### 기존 도구의 한계
| 도구 | 한계 |
|------|------|
| 캘린더 "시간 찾기" | 가능/불가 **이진법**만. 선호도·필수/선택 가중치 없음 |
| Doodle / when2meet | 캘린더에 이미 있는 걸 **사람이 또 입력**. 모두를 동등 취급. 주최자는 격자를 눈대중 |

→ 빠진 것은 **뉘앙스**. 현실의 조율은 free/busy가 아니라 "되는데 싫어 / 얘는 빠져도 돼"의 게임이다.

## 3. 핵심 설계 원칙 (Spine)

> **일정 조율의 진짜 작업은 "빈 시간 찾기"가 아니라 "서로의 사정을 협상하기"다. 그 협상이 메신저로 새는 순간 도구는 무력해진다. 그래서 dayfix의 모든 설계는 '협상을 제품 안에 붙잡아두기' 위한 것이다.**

아래 모든 정책은 이 한 문장의 따름정리다.

---

## 4. 솔루션 & 정책

### 4.1 가용성은 3단계 (이진법 아님)
`가능` / `조율 가능` / `불가`

- **왜:** 이진법은 협상이 일어나는 *회색지대를 지워서* 메신저로 내보낸다. "조율 가능"은 "되긴 하는데 사정이 있다"는, 지금 DM으로 하는 바로 그 말을 제품 안에 1급 시민으로 앉히는 상태다.

### 4.2 "조율 가능"은 조건 + 메모를 동반
조율 가능을 고르면 → **슬롯 맥락에 맞는 체크박스(예측) + 주관식**을 노출한다.

- **왜 체크박스인가:** 자유 메모는 "기록"이지만 체크박스는 **계산 가능한 데이터**다. "오후면 돼요"를 체크하면 그 사람의 오후 칸을 자동으로 `가능`으로 반영할 수 있다. 협상이 점수·매트릭스에 바로 들어온다.
- **예측은 슬롯 기준:** 후보 시간이 전부 오후인데 "오전이면 가능"을 띄우면 노이즈. 슬롯 시간대·과거 패턴에 맞는 것만 5~6개 노출 + 항상 "직접 입력"(주관식).
- **주의:** 체크박스는 앵커링이 있다 → 보기는 짧게, 주관식을 동등하게 보이게.

### 4.3 필수 / 선택 참여 + "선택 = 후순위" 정책
참석자는 `필수(required)` 또는 `선택(optional)`.

- **정책:** 선택참여자의 사정은 날짜 선정 시 **후순위로 반영**한다. 점수 가중치를 낮춰, 선택자의 불가가 회의 성사를 막지 못하게 한다("참고 자료처럼" 반영).
- **체면 처리:** 선택참여로 빠지는 행위는 **저소음·정상(normalized)** 이어야 자유롭다. 공지성 알림(❌ "OO님이 빠졌습니다") 금지, 매트릭스에서 조용히 분리 표시(✔). 강등이 아니라 "이 회의에 대한 관여도"일 뿐.
- **합의:** 선택참여자는 눈치 없이 자유롭게 — 단 위 저소음·정상화 조건이 충족될 때.

### 4.4 점수 알고리즘
```
ROLE_WEIGHT  = { required: 1.0, optional: 0.25 }
AVAIL_VALUE  = { 가능: 1.0, 조율가능: 0.6, 불가: 0.0 }

slot_score = Σ_member( AVAIL_VALUE[응답] × ROLE_WEIGHT[역할] ) / Σ_member( ROLE_WEIGHT[역할] ) × 100
```
- 동점이면 "필수 전원 가능"을 우선.
- 효과 검증: 필수 전원 가능 + 선택 1명 불가 슬롯이, 모두 가능하지만 필수가 조율 상태인 슬롯보다 위로 온다 → "선택 = 후순위"가 그대로 작동.

### 4.5 화면 2층 — 결정 + 이해
- **1층 · 추천 랭킹 (결정용):** 점수순 슬롯. 점수·배지·"누가 왜 걸리는지" 한 줄. 안 읽어도 1번 누르면 끝.
- **2층 · 가용성 매트릭스 (이해/검증용):** 시간 × 사람. 칸 색으로 3단계, 호버 시 조건·메모. "왜 이 시간이 1등이지?"를 납득시킨다.
- 사용자의 요구("동료들의 상황을 파악하고")가 매트릭스의 존재 이유다.

### 4.6 입력 마찰 최소화
- **캘린더 프리필:** 참석자 화면은 빈 격자가 아니라 *이미 채워진 상태*. "틀린 것만 바꾸면 돼요."
- **반박 모델:** when2meet식 "빈 칸 채우기"가 아니라 "추천을 반박하기".
- **상시 소프트 선호:** "나는 1~2시 회의 싫어요"를 한 번만 등록 → 회의마다 재입력 제거. (향후)

### 4.7 전원 불가 시 — "거래 제안" 정책
모두 1시간이 안 되면 **절대 "결과 없음"으로 끝내지 않는다.** 대신 거래를 제안:
- "전원 불가 → 필수 4명만 맞추면 화 10시 / 선택 1명 빼면 수 2시"
- "한 명이 15분 양보하면 목 4시"
(향후 구현)

## 5. 정보 구조 & 화면

### 5.1 메인(홈) = 캘린더 통합
dayfix는 캘린더 통합 서비스다. 홈은 두 축이 공존한다.

```
┌─ 예정된 일정 (무조건) ───────┬─ 조율 (액션) ─────────────┐
│ 오늘                        │ 🔔 새 조율 알림            │
│  14:00 스프린트 킥오프 회의A │   민정 → 'X' 응답 필요      │
│  16:30 1:1            Meet  │ ───────────────────────── │
│ 이번 주                     │ 🔴 내 응답 대기 (2)        │
│  수 10:00 디자인 리뷰        │ 🟡 확정할 수 있어요 (1)     │
│  목 16:00 ...               │ ⏳ 조율 진행 중 (3/6)       │
└─────────────────────────────┴──────────────────────────┘
        ↑ 캘린더(읽기 전용)            ↑ dayfix 고유 액션
```
- **예정된 일정:** 통합 캘린더에서 끌어온 확정 일정. 열자마자 "오늘/이번 주" 오리엔테이션.
- **조율 알림 + 목록:** 새로 조율할 일정의 *알림*(조율 시작/확정/변경 등 이벤트성)과 *목록*(내 응답 대기 / 확정 대기 / 진행 중).

### 5.2 우선순위 모델 — 2축
정보는 **"누가 막혔나(내가 막고 있나) × 시간이 임박했나"** 로 줄 세운다.
1. **내 차례** (응답 대기 / 확정 대기) — 내가 남을 막고 있음 = 누수 발원지. 나만 풀 수 있음.
2. **임박한 확정 회의** — "20분 뒤 회의 > 금요일까지 줄 응답". 시간 임박이면 최상위로.
3. **다가오는 일정 / 진행 중 조율** — 멀리 보기·현황 파악.

### 5.3 상세 = 조율 화면
홈의 카드 클릭 → **조율 상세**(= 현재 구현된 화면: 추천 랭킹 + 가용성 매트릭스 + 응답 드로어).
네비게이션: **홈(캘린더+조율) → 상세(조율 작업)**.

### 5.4 경계 — 보여주되 캘린더가 되지는 않는다
월간 그리드·반복 일정·드래그 편집은 구글 캘린더 영역. dayfix의 "예정 일정"은 **읽기 전용 오리엔테이션**이고, 손으로 *작업*하는 건 조율(랭킹·매트릭스·응답)이 주인공.

## 6. 약한 고리 & 대응

| 리스크 | 대응 |
|--------|------|
| "조율 가능"이 만능 도피처(조건 안 적음) → 회색지대가 다시 블랙박스 | 조율 가능 시 **조건(체크박스/메모) 반강제** |
| 자발적 후순위의 역설(먼저 누르기 눈치) | 후순위를 "빠짐"이 아니라 **"유연하게 맞출게요" 긍정 신호**로 라벨링 |
| 빈 인박스(액션 0건이면 홈이 허전) | 섹션 비면 **숨김** + 강한 빈 상태("다 정리됐어요 👏") |
| 예정 일정 범위 | 캘린더 통합이므로 전체 캘린더 지향(가능 범위 내) |

## 7. 결정 로그

### 확정
- dayfix = **캘린더 통합** 기반 사내 회의 조율 서비스.
- 가용성 **3단계**(가능/조율가능/불가).
- "조율 가능" = **슬롯 기반 체크박스 + 주관식**.
- **선택 참여 = 후순위**(알고리즘 가중치 + 화면 분리 표시).
- 화면 **2층**(랭킹/매트릭스), 네비 **홈 → 상세**.
- 홈은 **예정 일정 + 조율 알림·목록** 공존.

### 보류/향후
- 상시 소프트 선호 1회 등록.
- "전원 불가" 거래 제안.
- 알림 표면(조율 시작/확정/변경) 상세 설계.

### 열린 질문
- 알림의 채널(인앱만? 이메일/슬랙 연동?).
- "확정" 시 캘린더 초대 발송 흐름.

## 8. 디자인 시스템 정책
- **단일 기준(SSOT):** thaki TDS(`@thakicloud/shared`). 색·타이포·컴포넌트는 TDS 토큰만 사용, 하드코딩 금지.
- **일관성 최우선:** 개발 완성도보다 디자인 일관성을 우선(프로토타입 단계).
- TDS에 없는 요소(예: Avatar)만 토큰 기반으로 직접 구현.

### 8.1 컬러 규칙 (확정)
TDS 기본 팔레트를 **그대로** 사용한다. 색은 의미를 갖는다.

| 색 | 의미 | 쓰는 곳 |
|----|------|---------|
| **블루** (Primary) | 주요 액션 · 긍정 | 새 회의 만들기, 확정, 추천 1위 강조, 주요 CTA |
| **초록** (success) | 가능 · 긍정 상태 | 가용성 '가능', 캘린더 연동됨 등 |
| **앰버** (warning) | 조율 가능 · 주의 | 가용성 '조율 가능' |
| **레드** (error) | 불가 · 위험 | 가용성 '불가', 삭제 |

> 검토 후 폐기: "무채색 크롬 + 블루=긍정만"(Notion 스타일) 방향을 시도했으나, **블루가 주요 액션·긍정으로 잘 작동**해 원안(TDS 기본)을 유지하기로 확정. 다시 무채색을 제안하지 말 것.
